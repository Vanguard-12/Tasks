# Планирующий агент на LangGraph

## Описание

Этот репозиторий содержит минимальный пример **LangGraph**‑агента, который сначала планирует задачу, а затем последовательно исполняет каждый шаг плана, пока не получит окончательный результат.

## Требования

- Python **3.10** или выше
- `pip`

## Установка

```bash
# склонируйте репозиторий
git clone <repo‑url>
cd <repo‑directory>

# создайте виртуальное окружение и активируйте его
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

# установите зависимости
pip install -r requirements.txt
```

> **Примечание:** Для работы LLM используется `langchain-ollama`. Убедитесь, что у вас запущен Ollama‑сервер и доступна модель `llama3` (или замените модель в коде).

## Запуск примера

```bash
python agent.py
```

По умолчанию исполняется задача:

```
Сравни Python и JavaScript
```

Вывод выглядит примерно так:

```
Задача: Сравни Python и JavaScript

План:
1. Характеристики Python
2. Характеристики JavaScript
3. Сравнение областей применения
4. Вывод

[Шаг 1] Python — высокоуровневый язык с динамической типизацией, широким набором библиотек и акцентом на читаемость кода.
[Шаг 2] JavaScript — язык сценариев, работающий в браузере и на сервере (Node.js), обладает асинхронной моделью исполнения и тесно связан с веб‑технологиями.
[Шаг 3] Python обычно используют в науке о данных, автоматизации и бэкенде; JavaScript — в веб‑разработке и интерактивных UI.
[Шаг 4] Итог: Python удобнее для быстрого прототипирования и аналитики, а JavaScript незаменим для клиент‑сайда и интерактивных веб‑приложений.

Итоговый вывод: Python и JavaScript решают разные задачи; их выбор зависит от контекста проекта.
```

## Структура проекта

- `agent.py` – реализация агента: состояния, узлы, граф и точка входа `main()`.
- `requirements.txt` – список зависимостей.
- `README.md` – инструкции и пример вывода.
- `.gitignore` – стандартные исключения (Python, виртуальные окружения и т.д.).

## Как это работает

1. **planning** – LLM генерирует план из 3‑6 шагов и сохраняет его в состоянии.
2. **execution** – каждый вызов берёт один шаг из плана, передаёт его LLM и сохраняет результат.
3. **should_continue** – проверяет, закончились ли шаги, и направляет поток либо к следующему `execution`, либо к `finish`.
4. После завершения все результаты объединяются и передаются LLM для получения финального резюме.
